## INPUTS

Alert Radar receives raw webhook payloads from external alert systems. There is no fixed format — payloads vary by source and must be normalized before routing.

**TradingView webhook (common format):**
```json
{
  "ticker": "ETHUSDT",
  "close": "3198.50",
  "volume": "182450",
  "time": "2026-04-15T09:00:00Z",
  "alert_name": "BO_ETH_4H_LONG",
  "message": "ETH breakout above 3200 on 4H"
}
```

**Custom script alert (structured format):**
```json
{
  "asset": "BTCUSDT",
  "alert_type": "BREAKOUT",
  "zone_id": "Z001",
  "timeframe": "4H",
  "direction": "LONG",
  "price": 71250.00,
  "timestamp": "2026-04-15T09:15:00Z",
  "source": "euru_morning_scan"
}
```

**Internal references required at runtime:**
- `WATCHLIST_OFICIAL.md` — list of approved assets for routing
- `LISTA_PROIBIDA` — list of blocked assets (absolute rejection)
- `cooldown_config` — per-timeframe cooldown windows (default: 60 min for 4H, 240 min for 1D)

---

## OUTPUTS

A **Normalized Alert** with routing decision. See OUTPUT_FORMAT.md for full schema.

**Status values:** `ROUTED`, `REJECTED`, `OBSERVATION_ONLY`, `DUPLICATE`, `FREEZE_BLOCK`, `INVALID`

---

## NORMALIZATION MAPPING

Regardless of source format, the output normalized alert must conform to:

```yaml
euru_alert:
  alert_id: string           # UUID generated at receipt
  received_at: string        # ISO-8601 UTC of receipt
  source: string             # TRADINGVIEW | CUSTOM_SCRIPT | MANUAL
  asset: string              # normalized symbol, e.g., "ETHUSDT"
  alert_type: string         # BREAKOUT | SETUP | STRUCTURE | REGIME
  direction: string          # LONG | SHORT | NEUTRAL
  timeframe: string          # 1D | 4H | 1H | 15M
  trigger_price: float
  trigger_timestamp: string  # ISO-8601 UTC of triggering event
  zone_id: string | null
  age_candle_periods: float  # (received_at - trigger_timestamp) / candle_duration
  raw_message: string        # original unmodified message
```

## VALID SITUATIONS

**Scenario A — Clean route:**
- TradingView fires ETHUSDT 4H breakout alert. Symbol is on watchlist, not on LISTA_PROIBIDA, payload complete, age = 8 min (< 1 period), no duplicate in cooldown. Mode = SIMULATE.
- Result: `ROUTED` → Structure Hunter

**Scenario B — READ_ONLY mode:**
- Same alert, but system_mode = READ_ONLY.
- Result: `OBSERVATION_ONLY` — alert logged to Journal Learning, pipeline not activated.

**Scenario C — LISTA_PROIBIDA hit:**
- Alert arrives for "XYZUSDT". Asset found on LISTA_PROIBIDA.
- Result: `REJECTED` — log `ASSET_PROHIBITED`. Flag as security event.

**Scenario D — Duplicate suppression:**
- ETHUSDT BREAKOUT LONG 4H alert at 09:00. Identical alert arrives at 09:40. Cooldown = 60 min.
- Result: `DUPLICATE` — second alert suppressed, not routed.

---

## INVALID SITUATIONS

**Missing alert_type:**
- Payload arrives with symbol and price but no discernible alert type (not parseable from message).
- Result: `INVALID` — cannot route without alert_type. Return error.

**Stale alert:**
- Alert for BTCUSDT 4H arrives with timestamp 9 hours ago (= 2.25 candle periods).
- Result: `REJECTED` — `ALERT_STALE`. Threshold exceeded.

**Attempting to add trade signal during normalization:**
- Operator instructs Alert Radar to classify alert as "high confidence" based on message content.
- Invalid. Alert Radar normalizes format only. Confidence and quality assessment is downstream.

---

## NOTES

- Alert Radar is the **pipeline entry point** — no signal bypasses it.
- All alerts are logged to Journal Learning including REJECTED and DUPLICATE (for alert pattern analysis).
- `alert_id` is generated by Alert Radar at receipt time — it is the primary key for the alert throughout the pipeline.
- If both symbol validation AND LISTA_PROIBIDA check fail (edge case: same asset added to both lists), LISTA_PROIBIDA takes precedence: `REJECTED`.
